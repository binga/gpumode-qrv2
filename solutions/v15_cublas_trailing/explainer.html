New experiment: V15

Direct cuBLAS trailing updates

V15 keeps V13's successful double-precision shared-memory panel, then tests whether ATen tensor construction and batched matrix multiplies are the next removable bottleneck. The result is a useful negative experiment: correctness passes, but the direct path regresses medium shapes.

18.4ms Best logged V13 geomean
65.6ms V13 on batch=640, n=512
16 Panel/trailing steps for n=512
21.1ms V15 B200 7-shape geomean

Pipeline

1. Fused panel Factor each 32-column panel in shared memory with double reductions.
2. Build V/T Fill preallocated workspaces using CUDA kernels instead of ATen tensor ops.
3. Apply WY Use three row-major strided-batched cuBLAS calls except on high-batch n=512.
4. Dispatch fallback Keep fused n<=32, V13's high-batch n=512 path, and torch.geqrf for large shapes.

What changed from V13

  • Removed the unnecessary full-matrix clone before the small n<=32 path.
  • Preallocates V, T, W1, and W2 once per call.
  • Builds V and T on CUDA instead of repeated ATen slice/fill/copy calls.
  • Writes the final -V @ W2 update directly into the live trailing slice with cuBLAS beta=1.
  • Routes high-batch n=512 back through V13's ATen trailing path because direct cuBLAS was not robust there.

Risks to verify

  • Correctness: Popcorn B200 passes 22/22 after avoiding explicit streams and preserving V13 for high-batch n=512.
  • Performance: the serial T builder and pedantic FP32 cuBLAS path regress n=176 and n=352.
  • Promotion: V13 remains the best confirmed configuration until trailing update fusion avoids the T builder bottleneck.

Validation Target

V15 passes Popcorn B200 correctness and benchmarks at 21.1ms 7-shape geomean versus V13's 18.4ms. The direct cuBLAS idea should not be promoted as-is. The next useful follow-up is either a parallel T builder or a truly fused trailing update that never materializes V, T, W1, and W2 as separate global-memory workspaces.

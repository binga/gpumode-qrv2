V1: Blocked Householder optimization

Panel factorization via torch.geqrf + WY trailing update via batched GEMM.

Key Idea

Instead of applying each Householder reflection immediately to all remaining columns (n rank-1 updates), we accumulate a block of nb=32 reflections and apply them in one shot as a matrix multiply.

The rank-1 updates are memory-bound (low arithmetic intensity). The batched GEMM is compute-bound and can use tensor cores. This is the fundamental insight behind every fast QR implementation on GPU.

Algorithm: Block-by-Block

The Three Steps Per Block

1
Panel Factorization

Factor columns j to j+nb using torch.geqrf on the panel submatrix. This produces Householder vectors V and scalars τ. Still sequential column-by-column, but only nb columns instead of all n.

2
Build WY Compact Form

Accumulate the nb reflections into the WY representation: the block reflector is I − V T VT where T is an nb×nb upper triangular matrix. T is built column-by-column — cheap since nb is small (32).

3
Trailing Update via GEMM

Apply the block reflector to the trailing matrix:
A[j:, j+nb:] −= V · T · VT · A[j:, j+nb:]
This decomposes into two batched matrix multiplies — the compute-bound heart of the algorithm. With TF32 tensor cores on B200, this runs at ~2× the throughput of FP32.

The WY Representation

The key data structure. Given nb Householder reflectors Hj = I − τj vj vjT, their product is:

H1 H2 ··· Hnb = I − V T VT

where V = [v1, v2, ..., vnb] and T is nb×nb upper triangular, built recursively:

T[j, j] = τ[j]
T[0:j, j] = −τ[j] · T[0:j, 0:j] @ V[:, 0:j]ᵀ @ V[:, j]

This converts nb sequential rank-1 updates into two GEMMs — the fundamental speedup.

Implementation

def _blocked_qr(A, nb=32):
    batch, n, _ = A.shape
    H = A.clone()
    tau_all = torch.zeros(batch, n)

    for j in range(0, n, nb):
        actual_nb = min(nb, n - j)

        # 1. Panel factorization
        panel = H[:, j:, j:j+actual_nb].contiguous()
        panel_h, panel_tau = torch.geqrf(panel)
        H[:, j:, j:j+actual_nb] = panel_h
        tau_all[:, j:j+actual_nb] = panel_tau

        # 2. Build V and T
        V = _build_V(panel_h, actual_nb)
        T = _build_T(V, panel_tau)

        # 3. Trailing update: H -= V @ T @ V^T @ trailing
        trailing = H[:, j:, j+actual_nb:].contiguous()
        W = torch.bmm(V.transpose(1, 2), trailing)  # GEMM 1
        W = torch.bmm(T, W)                          # small matmul
        H[:, j:, j+actual_nb:] = trailing - torch.bmm(V, W)  # GEMM 2

    return H, tau_all

Limitations

  • Panel still uses torch.geqrf — not a custom kernel. The panel is the sequential bottleneck and needs a fused CUDA kernel (→ V2).
  • V and T built with Python loops — overhead from tensor indexing and temporary allocations. Should be fused into C++/CUDA.
  • Extra memory allocations — clone(), contiguous(), V and T tensors created each block step.
What's next (V2): Replace the panel factorization with a custom CUDA kernel that processes nb columns in shared memory with cooperative threads, and fuse V/T construction into the kernel. The trailing GEMM stays as cuBLAS — it's already optimal.

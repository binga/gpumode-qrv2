What It Does

One-liner

Calls PyTorch's torch.geqrf(A) which wraps cuSOLVER/MAGMA's batched QR on CUDA. This is the competition's reference implementation — every submission is measured against it.

import torch
from task import input_t, output_t

torch.backends.cuda.matmul.allow_tf32 = True
torch.backends.cudnn.allow_tf32 = True

def custom_kernel(data: input_t) -> output_t:
    return torch.geqrf(data)

Algorithm: Householder QR

Given a square matrix A (n×n), QR factorization finds:

  • Q — orthogonal matrix (QTQ = I)
  • R — upper triangular matrix

such that A = QR.

torch.geqrf returns the compact Householder form: (H, τ) where R lives in the upper triangle of H, and Householder reflection vectors live below the diagonal. Q is reconstructed via torch.linalg.householder_product(H, τ).

Why It's Slow

Bottleneck: Sequential column processing

Householder QR processes columns one at a time. Each column requires:

  1. Compute column norm (reduction across rows)
  2. Form Householder vector
  3. Apply reflection to all remaining columns (rank-1 update)

Steps 1-3 must complete for column j before column j+1 starts. This sequential dependency means the GPU's massive parallelism is underutilized — most threads are idle waiting for the current column to finish.

The fix is blocked Householder QR: process nb columns at a time (the "panel"), then apply all nb reflections to the trailing matrix as a single large GEMM. The GEMM is compute-bound and uses tensor cores — this is where the 63× speedup comes from.

Custom CUDA kernel for panel factorization + cuBLAS GEMM for trailing update.

Architecture Split

Panel Factorization
Custom CUDA Kernel

Memory-bound, sequential within each column. Multiple threads cooperate on norms and reflector application via shared memory reductions.

One thread-block per batch matrix. 256 threads per block. Shared memory for partial sums.

Trailing Update
cuBLAS Batched GEMM

Compute-bound. Two batched matrix multiplies with TF32 tensor cores on B200. Already near-optimal via cuBLAS.

torch.bmm dispatches to cublasGemmStridedBatched with TF32.

Panel Kernel: Thread Cooperation Model

Thread block for one matrix (n=512, 256 threads): Column j processing: Step 1: Column norm (parallel reduction) ┌─────────────────────────────────────────────┐ │ T0: sum a[j+1]²+a[j+257]² → partial[0] │ │ T1: sum a[j+2]²+a[j+258]² → partial[1] │ │ ... │ │ T255: sum a[j+256]²+a[j+512]²→ partial[255]│ │ │ │ Reduce: partial[0] += partial[128] │ │ partial[0] += partial[64] ... │ │ → σ² = partial[0] │ └─────────────────────────────────────────────┘ Step 2: Householder vector (thread 0 only) ┌─────────────────────────────────────────────┐ │ v₀ = x₀ + sign(x₀)·√(x₀² + σ²) │ │ τⱼ = 2 / (v₀² + σ²) │ │ Store: a[j,j] = -sign(x₀)·‖x‖ │ └─────────────────────────────────────────────┘ Step 3: Normalize + apply (all threads) ┌─────────────────────────────────────────────┐ │ Normalize: a[i,j] /= v₀ (parallel) │ │ │ │ For each column k = j+1 .. col_end-1: │ │ dot = parallel reduce(vᵀ · A[:,k]) │ │ A[:,k] -= τⱼ · dot · v (parallel) │ └─────────────────────────────────────────────┘ __syncthreads() between each phase

Key Design Decisions

DecisionChoiceRationale
Threads per block min(256, ceil(n/32)*32) 256 threads handle n=512 with 2 rows per thread. Warp-aligned for efficient reductions.
One block per batch blocks = batch_size batch=640 saturates B200's 148 SMs (~4.3× occupancy). No need for multi-block per matrix.
Shared memory layout 2 × nthreads × float Two arrays: partial sums for norm, dot products for reflector application. 2KB for 256 threads.
Panel applies to panel only Columns j..j+nb Trailing columns updated via GEMM (next step). Panel kernel only touches its own nb columns.
TF32 in trailing GEMM allow_tf32 = True ~2× throughput. 10-bit mantissa is sufficient — tolerance is 20·n·eps32.

Compilation

The CUDA kernel is compiled at runtime via torch.utils.cpp_extension.load_inline. First call compiles (~10-30s), subsequent calls use the cached .so.

_kernel = load_inline(
    name="qr_v3",
    cuda_sources=[cuda_src],
    functions=["blocked_qr"],
    extra_cuda_cflags=["-O3", "--use_fast_math"],
    extra_ldflags=["-lcublas"],
)
Risk: Compilation time on Popcorn's B200 instances may eat into the 240s test timeout. The --use_fast_math flag trades strict IEEE compliance for speed — acceptable given the competition's generous tolerances, but could cause issues on near-collinear or clustered-scale inputs. Monitor test results.

Correctness Considerations

  • LAPACK convention: The kernel must produce the same compact format as torch.geqrf — R in upper triangle of H, Householder vectors below diagonal with implicit v[j]=1, tau as reflector coefficients.
  • Sign convention: r_jj = -sign(x₀)·‖x‖. Getting the sign wrong produces valid QR but doesn't match householder_product's reconstruction, causing the checker to fail.
  • Numerical stability: sigma_sq > 1e-30f guard prevents division by zero for zero columns. The v₀ computation uses the numerically stable form (add sign·norm instead of subtract).
  • TF32 in trailing GEMM: Only affects the trailing matrix update, not the panel factorization. Panel runs in full FP32 — this is where numerical precision matters most.

Expected Performance

Untested — requires B200

This version has not been benchmarked on B200 yet. Expected improvement comes from:

  • Eliminating PyTorch dispatch overhead in the panel (no Python-to-C++ crossing per column)
  • Cooperative thread parallelism for column norms and reflector application
  • TF32 tensor cores in trailing GEMM (~2× throughput over FP32)

The V/T construction still uses PyTorch tensor operations (Python loops, indexing). Fusing this into the CUDA kernel is the next optimization target.

What's next: Fuse V/T construction into the panel kernel. Use cuBLAS directly (via C++ handle) instead of torch.bmm. Size-adaptive dispatch: cuSOLVER for small n, this kernel for medium, multi-block cooperative for large.

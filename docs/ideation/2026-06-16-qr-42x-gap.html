Grounding Context

What we know about #1 (704μs): Must be using fully device-side QR with zero cuSOLVER/cuBLAS library calls. At 704μs geomean across 12 shapes including batch=640 n=512, they process ~57B FP32 flops in under 1ms. This implies tensor-core math (TF32 or FP16 accumulation) for the trailing update and a fused panel kernel that eliminates all inter-kernel memory traffic.

What's failed: Custom serial-column panel kernels (slower than cuSOLVER), custom V/T kernels (slower than ATen loops), nb=64 (correctness fails), TF32 (correctness fails on stressed matrices).

B200 specs: 227KB shared memory/SM, 108 SMs, ~2TB/s HBM bandwidth, ~180 TFLOPS FP32, ~720 TFLOPS TF32 tensor core.

Top Ideas (Ranked)

1. Cooperative Multi-Warp Panel with Column-Parallel Reflection

Panel factorization GPU parallelism

Instead of processing panel columns sequentially (one warp-reduce per column), use multiple warps cooperating on each column and overlap the norm computation of column j+1 with the reflection application of column j. Assign 8 warps (256 threads) per panel, with warps 0-3 handling the current reflection while warps 4-7 pre-compute the next column's norm. This halves the effective serial depth.

external: MAGMA's dgeqr2 batched kernel (Abdelfattah 2022) achieves 16-25× over cuBLAS using exactly this cooperative multi-warp panel with overlapped norm/reflection. The paper explicitly states "the panel kernels contribute significantly to total execution time" and "kernel fusion" of norm+reflection is the key optimization.

Expected: 5-10× speedup on panel-dominated shapes. For (640,512), panel factorization is ~40% of iteration time.

2. CUDA Graphs Capturing the Entire Block Loop

GPU parallelism Memory hierarchy

The (640,512) shape has fixed geometry — every submission sees the same batch/n. Capture the entire 16-iteration block loop as a CUDA graph during the first call: all kernel launches, cuBLAS calls, and memory operations become a single graph replay. Eliminates ~500+ individual kernel launch overheads (each ~5-10μs on B200) per QR factorization. The graph also enables the driver to optimize memory reuse and scheduling across iterations.

external: GPU MODE NVFP4 winners noted that kernel launch overhead dominated for memory-bound shapes. PyTorch's `torch.cuda.CUDAGraph` API captures arbitrary streams. The (640,512) shape repeats 200+ times in benchmark mode with identical dimensions — perfect for graph replay.

Expected: 2-5× from eliminating launch overhead. At ~500 launches × 5μs = 2.5ms overhead alone — comparable to the target 704μs!

3. CholeskyQR2 with Direct Compact-Householder Output

Algorithm alternatives GPU parallelism

Replace Householder QR entirely with CholeskyQR2: compute Q via A^T·A → Cholesky → back-solve, then re-orthogonalize once for stability. The critical insight: the algorithm is embarrassingly parallel — the GEMM A^T·A fully saturates the GPU (no column-by-column serial dependency). Convert the resulting Q,R back to compact Householder form (H,tau) via a single additional geqrf on R (which is nb×nb, trivially fast). Two GEMM + one tiny geqrf vs 16 sequential panel+trailing iterations.

reasoned: Householder QR's serial column dependency is the fundamental bottleneck — each column must wait for all prior reflections. CholeskyQR2 replaces this with 2× GEMM (fully parallel, tensor-core friendly) + Cholesky (trivial for n×n) + back-solve. The condition-number-squaring issue is handled by the "2" (two iterations). For well-conditioned matrices (most benchmark cases have cond≤4), even one iteration suffices.

Expected: 10-50× by converting from latency-bound serial algorithm to throughput-bound parallel GEMM. The A^T·A GEMM for batch=640, n=512 is a batched 512×512 SGEMM — exactly what tensor cores are designed for.

4. Fused Panel Kernel with Shared-Memory Ring Buffer

Panel factorization Memory hierarchy

Write a single CUDA kernel that handles the entire blocked QR for one batch matrix. The panel (m×nb=512×32) lives in shared memory. After factoring the panel in-place, the kernel does not exit — it directly applies the WY trailing update by streaming trailing matrix tiles through shared memory in a ring buffer pattern. V and T never leave shared memory. Each thread-block handles one batch matrix's entire QR, making (640,512) = 640 thread-blocks running in parallel.

direct: MAGMA paper Section 8 "Big Picture": "The fused panel and update kernels can be used to factorize the entire matrix, without utilizing the batch dgemm kernel." For n≤512 with 64KB panels fitting in B200's 227KB SMEM (leaving 163KB for trailing tiles), this is feasible.

Expected: 10-20× by eliminating all inter-kernel global memory traffic and cuBLAS overhead. All 16 block steps fuse into one persistent kernel per batch matrix.

5. Mixed-Precision Panel (FP16 Reflection + FP32 Accumulation)

Panel factorization Trailing update

Compute Householder reflections in FP16 for the dot products and column updates (2× throughput, half the SMEM footprint) but accumulate norms and tau in FP32. The trailing GEMM uses TF32 tensor cores. Key insight: the output must be FP32 but internal computation need not be — as long as the final H,tau values satisfy the residual tolerance. The tolerance is rtol=20·n·eps32 which allows significant intermediate rounding.

external: MixedPrecisionBlockQR (GitHub: jaidonlybbert) demonstrates FP16 trailing GEMM with FP32 panel achieving full FP32-quality output. The NVFP4 competition winners showed that mixed-precision with careful accumulation can match full-precision results. V3's experiment showed TF32 trailing failed on stressed matrices — but that was with nb=64; nb=32 + FP16 panel with FP32 accumulation at critical steps may pass.

Expected: 2-4× from doubling compute throughput and halving SMEM usage (allowing larger tiles or higher occupancy).

6. Direct cuSOLVER Batched API (Bypass ATen Dispatch)

Panel factorization GPU parallelism

Call cusolverDnSgeqrfBatched directly with array-of-pointers interface instead of at::geqrf. ATen's geqrf may not use the optimal batched path for our specific shapes — it might dispatch to per-matrix geqrf sequentially. The array-of-pointers batched API processes all 640 matrices in a single library call with internal parallelism.

reasoned: `at::geqrf` on a 3D tensor may internally loop over batch elements calling single-matrix geqrf, or may use cuSOLVER batched. The API choice matters for batch=640 — the batched version launches one internal kernel for all matrices. Worth verifying which path ATen takes and forcing the optimal one.

Expected: 2-5× if ATen is not already using the batched path. Easy to verify.

7. Tall-Skinny QR via Communication-Avoiding TSQR

Algorithm alternatives GPU parallelism

For the block panels (512×32), use TSQR (Tall-Skinny QR): split the 512-row panel into 16 tiles of 32×32 each, QR-factor each tile independently (fully parallel!), then merge pairwise in a reduction tree. Each 32×32 tile factorization uses the proven V11 fused kernel. The tree merge is log2(16)=4 levels of small QR on 64×32 matrices.

external: TSQR (Demmel et al. 2012) is the communication-avoiding algorithm for tall-skinny QR. It replaces the n-step serial Householder with a log(m/n)-deep reduction tree. For 512×32: 16 parallel 32×32 factorizations + 4 levels of 64×32 merges. Each V11's fused kernel handles the 32×32 base cases in ~6μs.

Expected: 5-8× on panel-dominated portion by parallelizing the inherently serial panel factorization.

Additional Promising Ideas

8. Warp-Specialized Persistent Kernel (Producer-Consumer)

Memory hierarchy GPU parallelism

Use Blackwell's warp specialization: dedicate 4 warps to panel factorization (producer) and 4 warps to trailing update (consumer), communicating through shared memory barriers. The producer completes a panel column and signals the consumer, which immediately starts applying reflectors to trailing columns while the producer moves to the next column. Pipelined execution hides panel latency behind trailing compute.

external: Modular's Blackwell GEMM blog shows warp specialization achieving 85% of SOTA. GPU MODE NVFP4 winners used dedicated TMA and tensor core warps. Same pattern applies: panel warp produces V/tau, trailing warp consumes.

9. Per-Shape Kernel Specialization via Template Metaprogramming

GPU parallelism Panel factorization

Since the benchmark shapes are known (only 7 unique n values), compile 7 specialized kernels at module load — each with n, nb, thread count, and SMEM layout hardcoded as template parameters. The compiler unrolls the inner loops, eliminates conditionals, and maps directly to registers. The Python dispatch is a simple switch on n.

reasoned: The fused_qr_small kernel in V11 uses runtime `n` — the compiler can't unroll the column loop. Template-specializing for n=32, 176, 352, 512 lets the compiler generate perfectly tuned code per shape. The compilation cost is 7×, but within the 240s budget (V11's single kernel compiles in ~18s).

10. Batched 3-Stage Pipeline: Panel → V/T → Trailing (Overlapped)

GPU parallelism Trailing update

For (640,512) with 16 block steps, pipeline across batch elements: while batch[0..213] is in the trailing update of step k, batch[214..427] is in V/T construction of step k, and batch[428..639] is in panel factorization of step k+1. Three concurrent streams, each occupying 1/3 of the SMs. Eliminates the serial dependency between panel and trailing across different batch elements.

reasoned: The serial dependency (panel→V/T→trailing) exists within each batch matrix, but different batch elements are independent. With batch=640 and 108 SMs on B200, we have enough parallelism to pipeline stages. Multi-stream execution overlap is native to CUDA.

Rejected Ideas (with reasons)

torch.compile for the block loop

V7/V8 proved this times out (>300s compilation). The compiler can't fuse across geqrf boundaries anyway.

nb=64 for fewer iterations

V10 proved this fails correctness on ill-conditioned matrices. The TF32 error accumulates across larger GEMM inner products.

Pure register-based kernel for n=512

512 floats per row × 512 rows = 1MB; register file is ~256KB per SM. Physically impossible without blocking.

Triton kernel

Triton doesn't support Householder QR's control flow (column-by-column dependencies with conditional pivoting). Its block-oriented model doesn't map to the reduction-heavy panel factorization pattern.

cuSOLVERDx

Probed on Popcorn B200 — headers not installed. Not available in the competition environment.
